Big-O NOTATIONS

It doesn't matter where you live, what company you work in, even if you are programming for over 15 years from now, but this is a thing that will surely chase you, either be it in a different term or in a different meaning. It is the thing that truly makes a better developer.


The Picture

"Big O", that I came to know about when I was in 11th Grade. It was not much of my interest since I was programming just for purpose of love. Programming is the only thing that you can do whatever you want it to do.

At the end of 11th grade, I fell in love with PHP and ready to dump JAVA. Looping through 100 items in one go was really easy for me. Searching for an item from dozens of array was just a breeze but thing to notice here that it was all happening on my Local Server (localhost).

When I hosted my project, it tends to take time to search for an item from a list of just 23 items. My thought was blaming the internet connection speed (first time effect of a developer) as it takes time to load the page, render it and then displaying items search results. Yes, many of you might question about optimization and I would like to say "My code was optimized", but was it really?

This is the turning point where I understood the importance of Data Structure and how algorithms can be used to speed up time consuming process and the biggest thing is, I always come around to dash into that Big O term.


Harsh but True

Any Coder can solve a problem but what matters is how well the problem is solved. This is where Big O can help us to know how well the problem is solved.

Now I started blaming my code rather than my Internet Speed.



Coding

Coding are the instructions given to a computer.

We write codes which are the instructions for a computer to perform a task and produce some sort of result as a output.

There are various ways of solving a same problem through coding. But, the code that is used to solve a problem is efficient or not?


Good Code and Bad Code

Two factors comes in for impression of a Good Code.

  1. Readability
  2. Scalability

Let's explain what are these 2 things :

  1. Readability:
    Commented and Documented Code appears clean and beautiful so that others can understand it. Even you as a programmer, working with so many languages at once can understand when you come to work with that same code after a long time.
  2. Scalability:
    Code that can be scaled. The Idea of scalability can be measured with Big O notation. The code that one could be familiar with because of Readability and then can be decided to change or extend it for more. In one word Code that can be scaled up.

Now you are ready to distinguish between Bad Code and Good Code.
